# Plausible analytics is server-side only, opt-in via env

WMGID supports optional Plausible analytics, gated on `PLAUSIBLE_DOMAIN` (the on-switch) with `PLAUSIBLE_HOST` for self-hosted deployments (defaults to `plausible.io`). When `PLAUSIBLE_DOMAIN` is unset, no analytics code runs and there is no third-party footprint at all.

All tracking — including pageviews — is fired **server-side** from the Hono handlers via Plausible's events API. There is no `<script>` tag in any view. The event set is deliberately small: a pageview on `/`, plus `Sign-in started`, `Sign-in cancelled`, `Sign-in success`, `Sign-in rejected`, and `Sign-in verify failed`, fired from the corresponding routes. No custom props are ever sent; no email, no `hd`, no `sub`, no OAuth state. The `url` field is built from `BASE_URL` plus a fixed path argument passed to the helper, so the raw request URL — which on the OAuth callback carries Google's `code` and `state` — can never reach Plausible.

The obvious alternative is Plausible's standard browser script. We rejected it for three reasons. First, the Rejected and VerifyFailed pages are HTML responses served *at* `/auth/google/callback?code=…&state=…`; a browser-side pageview would ship that URL — including the OAuth `code` — to Plausible. Server-side firing with a hand-built `url` removes that leak by construction. Second, the script would require widening the locked-down CSP (`script-src`, `connect-src`) for the Plausible host; the server-side path leaves CSP untouched. Third, the privacy story for a debugging/curiosity tool is materially better when "the browser never talks to Plausible" is literally true. We accept the trade-off that server-side events bypass Plausible's browser-based bot filtering and may slightly inflate counts; for this tool's volume that's tolerable and visible if it ever matters.

Client IP for outbound events follows the deployment chain — `CF-Connecting-IP` → leftmost `X-Forwarded-For` → `X-Real-IP` — reflecting the default Cloudflare → Dokku → Node topology. The selected IP and the inbound `User-Agent` are forwarded on the outbound call so Plausible can compute its daily-rotating salted visitor hash; Plausible does not store the raw IP. Tracking is fire-and-forget: failures are logged with `console.warn` and never block or fail the user-facing response.
